## 3. Test Cases

### T1 â€” Happy Path: End-to-End Async Report Generation

**File:** `MessagingReliabilityTest#happyPath_reportStatusBecomeDone`

**Steps:**
1. Login as `TENANT_ADMIN`, create a project.
2. `POST /projects/{id}/generate-report` with header `X-Correlation-ID: test-corr-<uuid>`.
3. Assert HTTP 202 Accepted; `jobId` present; `reportStatus=PENDING` in response body.
4. Assert `X-Correlation-ID` echoed in response headers.
5. **Awaitility** polls DB every 500 ms for up to 15 s:
   - `job.reportStatus == "DONE"`
   - `job.status == "COMPLETED"`
   - `job.correlationId == <sent correlationId>`
6. Assert exactly one `ProcessedMessage` row in DB.

**Pass condition:** DB updated correctly within 15 s.

---

### T2 â€” Idempotency: Duplicate Message Delivery

**File:** `MessagingReliabilityTest#idempotency_duplicateMessageSkipped`

**Steps:**
1. Create a Job row manually (`reportStatus=PENDING`).
2. Publish the same `ReportJobMessage` twice using the **same `messageId`** AMQP property via `RabbitTemplate`.
3. Awaitility waits for first delivery â†’ `reportStatus=DONE`.
4. Small delay to allow second delivery to be received.
5. Assert `processedMessageRepository.findAll().size() == 1` (second delivery was skipped).

**Pass condition:** Only one `ProcessedMessage` row; DB state not corrupted by redelivery.

---

### T3 â€” Correlation ID Auto-Generated When Absent

**File:** `MessagingReliabilityTest#correlationId_autoGeneratedWhenAbsent`

**Steps:**
1. `POST /projects/{id}/generate-report` **without** `X-Correlation-ID` header.
2. Assert 202 Accepted.
3. Assert `X-Correlation-ID` response header is non-blank (server-generated UUID).
4. Awaitility waits for `reportStatus=DONE`.
5. Assert `job.correlationId == <value from response header>`.

**Pass condition:** Auto-generated ID is propagated end-to-end.

## 5. DLQ / Retry Failure Scenario (Manual Verification)

The following scenario verifies that a poisoned message does not block
the queue and is safely routed to the DLQ after retry exhaustion.

**Steps:**
1. Publish a `ReportJobMessage` with a `jobId` that does not exist in the DB.
2. The consumer throws `IllegalStateException` and Spring AMQP retries 3Ă—
   (configured with `initial-interval=2s`, `multiplier=2`, `max-interval=10s`).
3. After max-attempts the message is NACK-ed; RabbitMQ routes it to `report.generate.dlq`.

**Verification:**
- RabbitMQ Management UI â†’ Queues â†’ `report.generate.dlq` â†’ **Get Messages**: message visible.
- Log lines: `[CONSUMER] Processing failed | messageId=... error=Job <id> not found`
- Metric: `workhub_report_messages_failed_total` incremented.
